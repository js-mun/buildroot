## SF, libdrm 관계
안드로이드의 유저 공간(User Space)과 커널 공간(Kernel Space) 사이를 이어주는 가교입니다.

- 하는 일: HWC가 "여기에 화면 버퍼를 배치해줘"라고 요청하면, HWC가 복잡한 커널용 ioctl 시스템 콜 구조체를 직접 짜지 않도록, drmModePageFlip(), drmModeAtomicCommit() 같은 편리한 C 언어 API 함수들을 제공해 줍니다.
- 커널로의 최종 전달: HWC 내부에서 libdrm 함수를 호출하면, libdrm은 내부적으로 /dev/dri/card0 같은 DRM 디바이스 노드를 통해 커널 내부의 DRM/KMS 드라이버에게 "화면을 다음 프레임으로 넘겨라(Page Flip)"라는 신호를 보냅니다.

```
+-------------------------------------------------------+
|                 안드로이드 앱 / UI 시스템               |
+-------------------------------------------------------+
                           │ (화면 버퍼 전달)
                           ▼
+-------------------------------------------------------+
|  [User] SurfaceFlinger (총감독: 레이어 수집 및 관리)     |
+-------------------------------------------------------+
                           │ (레이어 합성 요청: prepare/set)
                           ▼
+-------------------------------------------------------+
|  [User] HWC (HAL 라이브러리: 칩셋 하드웨어 제어 최적화)   |
+-------------------------------------------------------+
                           │ (DRM API 호출: Atomic Commit 등)
                           ▼
+-------------------------------------------------------+
|  [User] libdrm (표준 래퍼: ioctl 시스템 콜을 쉽게 변환)  |
+-------------------------------------------------------+
                           │
====================[ 커널 경계선 ]======================
                           │ (ioctl 호출: /dev/dri/card0)
                           ▼
+-------------------------------------------------------+
|  [Kernel] DRM/KMS 드라이버 (예: msm, exynos, i915)      |
+-------------------------------------------------------+
                           │ (물리적 주파수/신호 변환)
                           ▼
+-------------------------------------------------------+
|                    물리 디스플레이 패널                 |
+-------------------------------------------------------+
```



## Present(출력) 순간의 연결 고리: dma-buf와 HWC
SurfaceFlinger가 최종 합성된 레이어들을 모아 화면에 띄우라고 HWC(Hardware Composer)에게 명령을 내리는 순간(Present), 이 두 버퍼의 매핑 작업이 일어납니다.
```
[ SurfaceFlinger ]  ──( Present / 아웃풋 버퍼 전달 )──>  [ HWC (Hardware Composer) ]
                                                                 │
                                                       (dma-buf를 DRM FB ID로 변환)
                                                                 ▼
[ 커널 KMS 드라이버 ] <──( Atomic Commit / FB ID 전달 )────────┘
```
- dma-buf 내보내기: HWC는 SurfaceFlinger에게 받은 안드로이드 그래픽 버퍼에서 dma-buf 파일 디스크립터(FD)를 추출합니다.
- KMS 등록: 이 dma-buf FD를 커널 표준 DRM 함수를 통해 /dev/dri/card0에 등록 요청합니다. (이때 내부적으로 drmModeAddFB2WithModifiers 같은 API가 호출됩니다.)
- FB ID 발급: 커널은 이 메모리 영역을 검증한 뒤 "이 버퍼는 이제부터 KMS 내부 ID 42번이다" 하고 DRM FB ID를 HWC에게 돌려줍니다.


### 스캔아웃 (Scan-out)
이제 준비가 끝났습니다. SurfaceFlinger가 최종적으로 Present 함수를 마무리 짓고 디스플레이를 갱신할 때, HWC는 방금 발급받은 DRM FB ID를 커널 KMS의 물리적인 평면(Plane) 플러그에 꽂아 넣습니다.

현대 안드로이드는 100% Atomic KMS(원자적 화면 갱신) 방식을 사용하므로 아래와 같은 형태로 전달됩니다.
- HWC가 커널에 drmModeAtomicCommit() 시스템 콜을 호출합니다.
- 이때 인자로 "Plane 50번 장치에 DRM FB ID 42번 버퍼를 바인딩하고 화면을 갱신해라"라는 설정을 통째로 넘깁니다.
- 디스플레이 컨트롤러(하드웨어)는 다음 VSYNC(수직 동기화) 타이밍에 맞춰 비디오 메모리 속 FB ID 42번(사실상 앱이 그린 그 Gralloc 버퍼)의 픽셀 데이터를 읽어서 디스플레이 패널로 주파수 신호를 쏘아 보냅니다. (이 과정을 Scan-out이라고 합니다.)


### SF에서 present에 대한 fence signal을 받는 타이밍은 언제일까?

디스플레이 컨트롤러가 스캔아웃을 '완료(끝)'한 직후가 아니라, 다음 프레임의 스캔아웃이 '시작'되어 이전 버퍼가 화면에서 완전히 밀려난(Released) 직후입니다

안드로이드 그래픽스 아키텍처에서 이 시점을 명확히 이해하려면 Present Fence(또는 Retain Fence)와 Release Fence의 메커니즘을 매칭해봐야 합니다.

⏳ 펜스 시그널링의 정확한 타임라인
화면 주사율이 60Hz(1프레임 = 16.6ms)인 디스플레이를 기준으로, SurfaceFlinger(SF)가 보낸 버퍼 A의 펜스가 풀리는 과정을 시간 순서대로 뜯어보겠습니다.

```
[프레임 1] VSYNC ────────────────────────────────────────────────────────
   │
   ├─► SurfaceFlinger가 버퍼 A를 HWC/KMS에 전달 (Present 요청)
   │
[프레임 2] VSYNC (이 타이밍에 칩셋이 스캔아웃 시작) ───────────────────────
   │
   ├─► 디스플레이 하드웨어가 버퍼 A를 읽어서 화면에 뿌리기 시작 (Scan-out 진행 중)
   │   ※ 주의: 버퍼 A가 화면에 그려지는 중이므로, 여전히 하드웨어가 이 메모리를 물고 있습니다.
   │   ※ 따라서 이 프레임이 끝날 때까지 버퍼 A의 펜스는 절대 signaling 되지 않습니다!
   │
[프레임 3] VSYNC (다음 프레임 시작) ───────────────────────────────────────
   │
   ├─► 디스플레이 하드웨어가 다음 화면인 '버퍼 B'를 읽어서 뿌리기 시작
   │
   └─► [★ 이 순간 Fence Signal 발생!]
       이제 버퍼 A는 디스플레이 컨트롤러 손에서 완전히 해방(Released)되었습니다.
       커널 DRM/KMS 드라이버가 이 VSYNC 인터럽트를 받아 Fence를 Signal(Green Light) 상태로 바꿉니다.
```

💡 왜 스캔아웃 완료 직후가 아니라 다음 프레임 시작일까?
디스플레이 컨트롤러 하드웨어는 화면의 맨 위 픽셀부터 맨 아래 픽셀까지 위에서 아래로 순차적으로 픽셀 신호를 쏩니다(스캔아웃).

만약 마지막 줄 픽셀을 다 쐈다고(스캔아웃 완료) 해서 커널이 바로 펜스를 풀어버리면 어떻게 될까요? SurfaceFlinger나 앱이 "오, 버퍼 프리해졌네?" 하고 신나서 그 버퍼에 다음 그림을 마구 그리다가(Overwrite), 디스플레이 리프레시 타이밍과 미세하게 꼬여 화면이 찢어지는 티어링(Tearing) 현상이 발생할 수 있습니다.

안전하게 하드웨어가 아예 다른 버퍼(버퍼 B)로 시선을 돌려서 "더 이상 버퍼 A 메모리 주소는 쳐다보지도 않는 상태"가 되어야만 펜스가 풀립니다.
